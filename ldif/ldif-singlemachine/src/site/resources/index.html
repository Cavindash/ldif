1. About LDIF

The Linked Data Integration Framework is developed to answer the need of integrating RDF datasets from the Web. Data is represented on the Web of Linked Data using terms from a wide range of different vocabularies. Furthermore datasets usually define proprietary URIs to identify entities and statements made in different datasets can be conflicting when merging these datasets. In order to produce a consistent dataset by tackling these problems an integration framework has to offer at least these steps as shown in the following figure:



The LDI Framework sits in the "Data Access, Integration and Storage Layer". A typical workflow consists of the following phases:

  1. The Web Data Access Module obtains the relevant datasets for example by crawling the Web of Data or downloading specific dataset dumps. It associates the data with provenance information and then offers the data to the following phase.
  2. The Vocabulary Mapping Module translates the data to a target vocabulary chosen by the user to offer a uniform view on data from different datasets.
  3. The Identity Resolution Module identifies pairs of URIs that seem to correspond to the same real world object.
  4. The Quality Evaluation Module decides based on the data and on provenance information and background knowledge, how to resolve confliicting data values and produce a coherent view on the integrated data.

Outlook


The next steps we want to take to offer a scalable and configurable data integration solutions are:
  • Add Cluster profile: Processes and data are distributed over a cluster of machines (probably based on HDFS and Hadoop)
  • Support for modelling the integration flow: Currently the integration flow is static and can only be influenced by predefined configuration parameters.
  • Add Web Data Access Module
  • Add Quality Evaluation Module

2. Quick start


In order to experiment with LDIF follow these steps:
  1. Download the latest release (download link at the top of this page)
  2. Unpack the archive to an arbitrary location
  3. To run the command line tool type the following in the root directory of LDIF:
java -jar ldif-all-inclusive-TODO.jar ldif.local.Main <path-to-ldif-config-file>
For details about the configuration file and parameters, see section 5.

3. LDIF Components


The components or modules are the essential parts of the integration flow. These are the main parts that are wired together by the framework. Right now there is one implementation for the Mapping Module and one for the Identity Resolution Module. For the mappings we utilize the R2R Framework, for the identity resolution the Silk Framework is used. Implementations for the other modules will be added at a later time.

R2R


An overview and examples for mappings are given on the main page of the R2R Framework. The specification and user manual is described in a separate document.

Silk


An overview and examples can be found on the Silk main page.

5. Configuration options


In this section we describe how an LDIF configuration file looks like and which parameters you can modify to change the runtime behaviour of LDIF.

LDIF configuration file

LDIF is configured with an XML file, whose current structure is very simple because the integration flow is static at the moment - something that will change in future releases. A typical config file looks like this:

<LDIF>
  <Properties>test1.properties</Properties>
  <Sources>sources</Sources>
  <LinkSpecifications>linkSpecs</LinkSpecifications>
  <Mappings>mappings/mappings.ttl</Mappings>
  <Output>output.nq</Output>
</LDIF>

It has the following elements:

  • Properties: The path to a Java properties file for configuration parameters.
  • Sources: The directory to the source datasets. This is also the location where provenance data can be stored. These files must be in N-Quads format and may be compressed (.gz, .zip).
  • LinkSpecifications: The directory containing the Silk link specifications.
  • Mappings: File containing R2R mappings.
  • Output: The name of the output dataset with the integrated data.

All the paths are relative to the config file location. In this case there is a root directory with the config file and the test1.properties file in it. Furthermore the following directories would be nested in the root directory: linkSpecs, sources and mappings. As you can see, currently the datasets have to be in a local directory. More sophisticated data access will be implemented with the coming Data Access Module.

Configuration Properties

In the LDIF configuration file you can specify a (Java) properties file to further tweak certain parameters concerning the integration execution. Here is a list with all properties that can be set at the moment and the possible values for each property:

# Specify if all input quads should be included in the output file or only the mapped quads.
# Default: mapped-only

output = all | mapped-only

# Specify if URIs should be rewritten to a common URI in regard to the discovered links by the Identity Resolution Module.
# Default: true

rewriteURIs = true | false

# The graph containing the provenance information. These quads are only output at the end, but are not further processed by the integration flow.
# Default: http://www4.wiwiss.fu-berlin.de/ldif/provenance

provenanceGraphURI = http://www4.wiwiss.fu-berlin.de/ldif/provenance

# Here you can choose how data is stored while it is being processed.
# Regarding the space vs. speed compromise, there are two solutions right now: A pure in-memory solution and a solution based on Voldemort.
# Default: In-Memory

entityBuilderType = in-memory | voldemort

Setting the storage type:

Depending on your integration use case, especially the size of the input datasets, you may choose different storage types in order to increase execution speed or limit the memory foot print. Right now we offer two solutions:

  • An in-memory solution, which keeps most of the data in-memory to perform as fast as possible.
  • A file backed solution with Project Voldemort as storage solution.
Config property file changes:
entityBuilderType = Voldemort | In-memory

If this property is not set, it defaults to the in-memory solution.
